# プロジェクト管理ルール

## 開発フロー

### 作業前の確認
- `git status` で現在の状態を確認
- `npm run build` でビルドが通ることを確認
- 変更対象のファイルを必ず読んでから編集する

### コミットルール
- コミットメッセージは日本語で記述
- プレフィックスを使用: `feat:`, `fix:`, `refactor:`, `test:`, `docs:`, `style:`, `chore:`
- 1コミット = 1つの論理的な変更単位

### ブランチ戦略
- `main` ブランチに直接コミット（小規模プロジェクト）
- 大きな機能追加時はフィーチャーブランチを作成

## コード品質

### TypeScript
- strict モードを遵守
- `any` 型の使用は原則禁止（やむを得ない場合はコメントで理由を記載）
- 型推論が効く場合は明示的な型注釈を省略してよい

### React コンポーネント
- 関数コンポーネントのみ使用
- カスタムフックで状態ロジックを分離
- Props の型は interface で定義

### スタイリング
- Tailwind CSS のユーティリティクラスを優先
- カスタム CSS は `src/index.css` の @theme レイヤーに集約
- shadcn/ui コンポーネントを活用し、独自 UI の再発明を避ける

## 自律的な開発姿勢
- 判断に迷った場合は、既存コードのパターンに合わせる
- ビルドが壊れた状態でコミットしない
- タスクを小さく分割し、段階的に完了させる
